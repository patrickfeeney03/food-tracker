# Calorie Tracker: Schema Contract

This document defines the implementation-level data model agreed during pre-build planning. It is intentionally expressed as a schema contract rather than finished Drizzle code.

## Global Conventions

- Primary keys are UUID strings generated by the server.
- Audit timestamps are UTC instants stored as SQLite integers.
- Diary and goal dates are `YYYY-MM-DD` local calendar-date strings.
- The application timezone is initially `Europe/Dublin`.
- `PRAGMA foreign_keys = ON` is mandatory for every database connection.
- `PRAGMA journal_mode = WAL` is established and verified during database initialization.
- Integer `CHECK` constraints enforce positive quantities and non-negative nutrition values.
- All user-owned queries include `user_id`, including reads by primary key.
- JSON input is validated at the service boundary; Drizzle TypeScript types alone are not runtime validation.

## Unit Contract

### Energy

- UI unit: kilocalories (`kcal`).
- Storage unit: milli-kilocalories (`mkcal`).
- Example: `132.5 kcal` is stored as `132500 mkcal`.

### Nutrients

- UI unit: normally grams, with sodium optionally displayed in milligrams.
- Storage unit: milligrams (`mg`).
- Example: `4.7 g` protein is stored as `4700 mg`.

### Food Amounts

- Solid UI unit: grams; storage unit: milligrams.
- Liquid UI and storage unit: whole millilitres.
- Each food has an `amount_unit` of either `mg` or `ml`.
- A food's basis, serving, container, diary amount, and shortcut amount all use the food's `amount_unit`.

### Fixed-Point Parsing

- Decimal UI values are parsed from strings directly into integers; they are never routed through binary floating point before conversion.
- Portion counts use an integer fixed-point scale, initially thousandths. For example, `1.5` is stored as `1500`.
- Ratio results are rounded to the nearest integer mg or mkcal at the individual diary-entry boundary.
- `NUTRITION_MATH.md` is the source of truth for parsing, scaling, rounding, totals, and display precision.

## `users`

Purpose: local application identity and lightweight UI preferences.

Required fields:

- `id`
- `name`
- `email`
- `settings_json` for non-historical preferences such as dark mode
- `created_at`
- `updated_at`

Nutrition goals do not belong in `settings_json` because they are effective-dated.

## `auth_accounts`

Purpose: links a local user to Google.

Fields:

- `id`
- `user_id` Ôćĺ `users.id` with cascade on user deletion
- `provider`, fixed to `google` initially
- `provider_subject`, Google's stable account subject
- `email_at_link`
- `created_at`

Constraints and indexes:

- Unique `(provider, provider_subject)`
- Unique `(user_id, provider)`

Only the configured allowlisted Google identity may create/access the initial account.

## `sessions`

Purpose: revocable 90-day rolling login sessions for phone and desktop.

Fields:

- `id`
- `user_id` Ôćĺ `users.id` with cascade
- `token_hash`, never the raw cookie token
- `created_at`
- `last_seen_at`
- `expires_at`
- `revoked_at`, nullable
- `user_agent`, nullable display metadata

Constraints and indexes:

- Unique `token_hash`
- Index `(user_id, revoked_at, expires_at)`

## `nutrition_goals`

Purpose: preserves target history.

Fields:

- `id`
- `user_id` Ôćĺ `users.id` with cascade
- `effective_from` as `YYYY-MM-DD`
- `target_energy_mkcal`
- `target_protein_mg`
- `target_carbs_mg`
- `target_fat_mg`
- `created_at`
- `updated_at`

Constraints and indexes:

- Unique `(user_id, effective_from)`
- Index `(user_id, effective_from DESC)`
- Targets are non-negative

For diary date `D`, select the latest goal where `effective_from <= D`.

## `foods`

Purpose: mutable reusable food definitions. Existing diary logs do not calculate through these live values.

Identity and ownership:

- `id`
- `user_id` Ôćĺ `users.id` with cascade
- `name`
- `brand`, nullable
- `barcode`, nullable text

Measurement and label basis:

- `amount_unit`: `mg` or `ml`
- `basis_amount`: positive integer in `amount_unit`
- `serving_amount`, nullable positive integer in `amount_unit`
- `container_amount`, nullable positive integer in `amount_unit`

Nutrition on the stored basis:

- `energy_mkcal_per_basis`
- `protein_mg_per_basis`
- `carbs_mg_per_basis`
- `fat_mg_per_basis`
- `additional_nutrition_json`, nullable
- `notes`, nullable plain text for the user's reusable-food reference

Lifecycle:

- `created_at`
- `updated_at`
- `deleted_at`, nullable

`additional_nutrition_json` initially supports integer milligram values for:

- fibre
- sugar
- saturated fat
- sodium
- potassium

Notes belong to the reusable food only. They affect no nutrition calculation and are not copied into diary snapshots.

Constraints and indexes:

- Core nutrition is required and non-negative; zero is valid.
- `basis_amount > 0`.
- Optional serving/container values are null or positive.
- Index `(user_id, deleted_at, name)` for the active catalogue.
- Index `(user_id, deleted_at, brand)` as justified by measured search behavior.
- Partial unique active mapping for `(user_id, barcode)` where barcode is not null and `deleted_at` is null.
- A normalized brand/name match warns about a possible duplicate but is not unique.

## `diary_logs`

Purpose: authoritative historical food entries and nutrition snapshots.

Identity, destination, and provenance:

- `id`
- `user_id` Ôćĺ `users.id` with cascade
- `food_id` Ôćĺ `foods.id` with set-null on hard deletion
- `diary_date` as `YYYY-MM-DD`
- `meal_slot`: `breakfast`, `lunch`, `dinner`, or `snacks`
- `source_shortcut_id`, nullable provenance reference
- `shortcut_batch_id`, nullable UUID shared by one shortcut application
- `client_mutation_id`, nullable retry/idempotency key

Food snapshot:

- `food_name`
- `food_brand`, nullable
- `amount_unit`
- `basis_amount`
- `energy_mkcal_per_basis`
- `protein_mg_per_basis`
- `carbs_mg_per_basis`
- `fat_mg_per_basis`
- `additional_nutrition_per_basis_json`, nullable

Portion input snapshot:

- `portion_kind`: `unit`, `hundred`, `serving`, or `container`
- `portion_label`
- `portion_amount`
- `portion_count_milli`
- `resolved_amount`

Calculated entry totals:

- `energy_mkcal`
- `protein_mg`
- `carbs_mg`
- `fat_mg`
- `additional_nutrition_total_json`, nullable

Lifecycle:

- `logged_at` as the original UTC logging instant
- `created_at`
- `updated_at`
- `deleted_at`, nullable

Constraints and indexes:

- `resolved_amount > 0`.
- Snapshot nutrition and calculated totals are non-negative.
- Meal slot is restricted to the fixed four values.
- Unique `(user_id, client_mutation_id)` when the idempotency key is non-null.
- Dashboard index `(user_id, diary_date, meal_slot, deleted_at)`.
- Recency index `(user_id, food_id, deleted_at, logged_at DESC)`.
- Shortcut Undo index `(user_id, shortcut_batch_id, deleted_at)`.

All snapshot fields and calculated totals are written or updated in one transaction by the nutrition service. Editing a reusable food never updates these rows.

## `meal_shortcuts`

Purpose: named, reusable groups of foods with default exact amounts.

Fields:

- `id`
- `user_id` Ôćĺ `users.id` with cascade
- `name`
- `created_at`
- `updated_at`
- `deleted_at`, nullable

Indexes:

- `(user_id, deleted_at, name)`

## `meal_shortcut_items`

Purpose: ordered items within a meal shortcut.

Fields:

- `id`
- `shortcut_id` Ôćĺ `meal_shortcuts.id` with cascade
- `food_id` Ôćĺ `foods.id` without cascade
- `position`
- `default_amount`
- Optional default portion representation: kind, label, portion amount, and fixed-point count

Constraints and indexes:

- `default_amount > 0`.
- Unique `(shortcut_id, position)`.
- Index `food_id` to identify shortcuts blocked by an archived food.

Applying a shortcut validates that every referenced food is active, then inserts all diary snapshots inside one database transaction. No item is silently skipped.

## Data Deliberately Not Modelled Initially

- Food revisions
- Cooked-yield recipes
- Body weight
- Exercise calories
- Custom meal slots
- Multiple barcodes per food
- Offline mutation queues
- Import jobs
- Copy-day, copy-meal, or duplicate-entry operations
